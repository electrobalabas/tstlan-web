HTTP API
========

Health
------

``GET /health``
   healthcheck

Auth
----

``POST /auth/login``
   Принимает ``login`` и ``password``, создаёт серверную сессию, ставит cookie и
   возвращает пользователя с ``csrf_token``. CSRF-токена у клиента ещё нет,
   поэтому запрос защищён только проверкой ``Origin``; неверные учётные данные
   дают ``401``.

``POST /auth/logout``
   Отзывает текущую сессию. Требует ``X-CSRF-Token``.

``GET /auth/me``
   Возвращает текущего пользователя и CSRF-токен. Без действующей сессии - ``401``.

Users
-----

``GET /users``
   Список пользователей (``login`` и ``role``). Нужен форме совместного доступа,
   чтобы выбрать получателя из списка. Доступен любому аутентифицированному
   пользователю.

Devices
-------

Все маршруты приборов требуют действующей сессии, без неё - ``401``. Чтение
доступно любой роли, запись значений - только ``dev`` и ``admin``.

``GET /devices``
   Список приборов: идентификатор, имя, тип, статус и число переменных.

``GET /devices/{device_id}``
   Карточка прибора с перечнем переменных - имя, C-тип и режим доступа.

``GET /devices/{device_id}/values``
   Снимок всех читаемых переменных. Переменные mode ``W`` не возвращаются.

``GET /devices/{device_id}/values/{name}``
   Значение одной переменной. Для write-only переменной вернётся ``403``.

``PUT /devices/{device_id}/values/{name}``
   Запись значения. Роль ``user``, как и read-only переменная, вернёт ``403``;
   неверный тип - ``422``.

``GET /devices/{device_id}/history``
   Накопленная сервером история значений: список точек ``{t, values}``, где
   ``t`` - серверное время unix в секундах. Окно ограничено (~15 минут при
   шаге в секунду) и живёт в памяти процесса: перезапуск сервера историю
   обнуляет. Фронтенд сидирует этими точками графики при открытии карточки.

``GET /devices/{device_id}/stream``
   SSE-поток снимков значений. Каждое событие - объект ``{t, values}`` с
   серверным временем: история и поток ложатся на одну ось графика независимо
   от часов браузера. Аутентификация - по сессионной cookie: ``EventSource``
   не умеет ставить заголовки.

Configs
-------

``GET /configs``
   Список доступных пользователю конфигов с вычисленным доступом.

``POST /configs``
   Создание конфига. Публиковать ``PUBLIC`` могут только ``DEV`` и ``ADMIN``.

``GET /configs/{config_id}``
   Детальный конфиг.

``PUT /configs/{config_id}``
   Обновление. ``OWNER`` управляет метаданными и видимостью, ``WRITE`` может
   менять payload.

``DELETE /configs/{config_id}``
   Удаление, только владелец или admin через effective owner-доступ.

``POST /configs/{config_id}/shares``
   Выдать или обновить доступ другому пользователю. Доступно владельцу и
   администратору.

``DELETE /configs/{config_id}/shares/{login}``
   Отозвать share. Доступно владельцу и администратору.

Ошибки доступа
--------------

Сервисные исключения переводятся в HTTP-коды:

* ``401`` - нет действующей сессии или неверные учётные данные.
* ``404`` - прибор, переменная, конфиг или пользователь-грант не найдены.
* ``403`` - нет прав, нельзя читать write-only или писать read-only переменную,
  не пройдена проверка CSRF.
* ``422`` - некорректное значение или попытка поделиться с владельцем.
